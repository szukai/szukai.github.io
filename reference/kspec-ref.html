# Kloudless Specification

The Kloudless Specification (KSpec) file contains information describing the behavior, properties and format mappings/transformations of the connected service.

# KSpec Reference Layout

# parameter name

  • type: type
  • required/optional
  • sample: sample text
  • default: more sample text

# kloudless

  • type: string
  • required
  • sample: kloudless: connector

Each Kloudless Specification (KSpec) YAML file should begin with the kloudless key value pair. The kloudless key value dictates what kind of information is in the KSpec file and what structure the contents inside should be.

WARNING

The only acceptable field right now is connector. This will change as new features are implemented.

# version

  • type: string
  • required
  • sample: 0.2

Indicates what version this KSpec file is. As of right now the current version is 0.2.

# openapi

  • type: string
  • required
  • sample: connector-oapi.yml

Indicates the OpenAPI yaml file that contains the connected service's API defintion in the OpenAPI format.

It's also possible to embed the entire OpenAPI 2.0 spec in this section as well.

# script

  • type: string
  • optional
  • sample: connector-api.py

Indicates which custom connector python script file to refer to for routing and endpoint logic.

# service_info

  • type: mapping
  • required

The service_info section contains information for the connected service. It will be used to create new service record in the Kloudless' Services database.

It contains the following:

  • name
  • friendly_name
  • auth
  • apis
  • category
  • logo_url
  • initially_visible
  • visible

# name

  • type: string
  • required
  • sample: gdrive_demo

The internal name of the connected service. This name will be the name used inside the Kloudless Services database. It is recommended to use a simple or concise name.

WARNING

Duplicate names to existing Kloudless services e.g. gdrive are not allowed.

# friendly_name

  • type: string
  • optional
  • sample: Google Drive Demo

friendly_name will be the name shown in the account connecting page. It is highly recommended to make this name distinct for users to be able to recognize easily.

# logo_url

  • type: string
  • optional
  • sample: https://www.website.com/static/logo.png

The picture specified in the logo_url will be shown alongside the service name in the account connection page.

# auth

  • type: string
  • optional
  • sample: oauth2

Indicates what kind of authorization the connected service uses. Currently the supported authorization types supported are oauth2 and basic.

# apis

  • type: string/list
  • required
  • sample:
apis:
- storage
- events

Indicate what kind of API features the service supports. A service can support functionality from more than one API.

Currently Kloudless supports the following API:

  • storage
  • itsm
  • team
  • calendar
  • crm
  • sharing
  • events
  • messaging
  • construction
  • contact

The Kloudless Developer Documentation will have more information on each API category and their supported services.

# category

  • type: string
  • required
  • sample: storage

Indicates what main category the service can be classified as. Kloudless currently supports the following API categories:

  • storage
  • calendar
  • construction
  • messaging
  • crm
  • itsm

The Kloudless Developer Documentation will have more information on each API category and their supported services.

# initially_visible

  • type: boolean
  • optional
  • default: false
  • sample: true

Indicates if the service will be visible by default when it is first created. To be "visible" means the service is enabled to connect accounts. It is called "initially_visible" because it is only checked when this service is introduced to the database.

  • type: boolean
  • optional
  • default: false
  • sample: true

# authentication

  • type: mapping
  • required
  • sample:
authentication:
  handle_response: access_token
1
2

Contains information related to authentication. The content in this section will depend on what kind of authentication is specified inside service_info.

If using oauth, handle_response: access_token with the appropriately defined transforms should suffice.

If using basic authorization, a pre-auth mapping with the pre-authorization form fields is needed instead.

TIP

See the our guide on Authentication for more details.

# pre_auth

  • type: mapping
  • optional
  • sample: see authentication guide.

If the service is using OAuth2 authentication, then the pre-authorization related form fields should be specified here.

See the our guide on Authentication for more details.

# handle_response

  • type: string
  • required
  • sample: access_token

Indicates how the authentication response should be handled if the service is not using basic authorization.

See the our guide on Authentication for more details.

# Transforms

  • type: mapping
  • required
  • sample:
transforms:
  account_info: |
    account_name: {{data.email}}
    account_id: {{data.uid}}
1
2
3
4

The Kloudless specification file utilizes transforms to translate object data formats between the service endpoint API and the Kloudless platform. Transforms utilizes the Jinja2 template engine.

Check out the Transforms Guide for how to use transforms.

Transformation exceptions

There are a few differences between the KSpec transforms and the Jinja2 template. These exceptions are listed in the Transforms Exceptions section.
