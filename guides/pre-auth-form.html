# Customizing the Pre-Auth Form

During the authentication flow, the account and other required information are provided by the user. The Kloudless Custom Connector facilitates this process by rendering the pre-auth section from the Kloudless Specification into an HTML form.

# Pre-auth Form Informations

Kloudless will look for pre_auth in the authentication section in the Kloudless Specification (Kspec) file to render the form. Here's a quick example:

authentication:
    pre_auth:
        instructions: |
            <p>
              Please provide your Kapost account's domain, username, and password
            </p>
        fields:
            username:
                friendly_name: Email address
                input_type: email
                required: true
            password:
                friendly_name: API Token
                input_type: password
                required: true
            domain:
                friendly_name: Domain
                input_type: text
                required: true

The following fields should be defined:

  • instructions: Instructions on the necessary setups for the authentication process.
  • fields: A dict that contains key-value pair of name and information for each form field. See Form Fields.
  • stores_password: Optional. A boolean indicating whether Kloudless will store the password. This decides what password storing policy footnote is displayed. Defaults to false.

# Form Fields

It is possible to add multiple desired fields as name and information fields key-value pairs.

The available configurations for fields are as follows:

  • input_type: The input type of the field, e.g. email, url, password, text, url, checkbox.
  • friendly_name: The label for the field.
  • required: Boolean. Whether the field is required.
  • domainsuffix: Only used for subdomain field. See below for more information.

For the Basic Authentication flow to work, username and password are required in fields.

Note the following special fields:

subdomain

  • Requires an extra configuration, domainsuffix.
  • As an example, if the domainsuffix is set to '.abc.com', the resulting input in the pre_auth form looks like: [__subdomain__].abc.com

scheme

  • This field will only be rendered if subdomain field is also present.
  • The resulting inputs look like: [__scheme__]://[__subdomain__].abc.com

# Storing And Reusing

The form input data will be passed into account_info(). The Kloudless Connector will retrieve the authenticated user's information with the provided information. It can also store essential fields other than username and password in account's extra_data for later use. Check the Account Info section for further details.

The account's extra_data['url_vars'] is a reserved field in extra_data that Kloudless uses to render the request URL easily. This prevents the developers from having to specify the URL rendering in account_info. E.g., If you have URL defined as http://{domain}.kapost.com/, the user filled domain field will be formatted into the URL.
